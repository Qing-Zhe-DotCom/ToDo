# Codex 协作与代码规范（本仓库）

目标：让“格式与低阶错误”交给工具，让“架构与业务权衡”留给人；提高一致性、可维护性、可审计性。

适用范围：本仓库所有文件；子目录可用更深层 `AGENTS.md` 覆盖/补充。

## 1) 协作流程（强制）：6 子代理并行

任何任务开始后，**必须**启用 **6 个子代理**，主线程只做任务拆分、分派与最终汇总：

- 子代理总数：6
- 模型：`gpt-5.2`
- 推理强度：`xhigh`
- 角色分工（建议）：
  - 4 个 `worker`：实现/修改代码（每个 worker 负责互斥的写入范围，避免冲突）
  - 2 个 `explorer`：审计（只读检查：安全、正确性、边界条件、性能、可维护性、测试覆盖、回归影响）
- 主线程职责：
  - 澄清需求（必要时）
  - 拆分为 4 个并行实现子任务 + 2 个并行审计子任务
  - 集成最小必要改动并给出最终交付说明

若由于平台/权限限制无法创建子代理，主线程必须在回复中**明确说明原因**，并回退到“单线程实现 + 自审计清单”的方式继续推进。

审计子代理的最小清单（必须覆盖）：
- 模块边界（`module-info.java` 的 `exports/opens/requires` 是否被不必要扩大）
- UI 线程与资源释放（JavaFX 线程、数据库连接/statement/resultset 关闭）
- 安全与隐私（SQL 注入、日志泄漏敏感信息、路径/输入校验）
- 回归影响（API 行为变化、数据兼容性、配置迁移、测试缺失）

## 2) 工程结构与边界（以现有包为准）

保持现有分层与依赖方向：UI →（业务）→ 数据；禁止反向依赖。

- `com.example.application`：应用启动/生命周期编排（不写业务逻辑）
- `com.example.controller`：UI 事件编排层（不写 SQL；不做重计算；不阻塞 UI 线程）
- `com.example.view`：可复用 UI 组件/视图（尽量无业务；复杂逻辑放到业务层）
- `com.example.model`：领域模型（尽量 UI 无关；不持有全局单例；谨慎引入 JavaFX Property）
- `com.example.data`：数据访问/仓储（DAO/Repository；不依赖 UI 包）
- `com.example.databaseutil`：底层数据库工具（连接、初始化、迁移、通用 SQL 工具；不夹杂业务）
- `com.example.config`：配置与装配（不做业务判断）

若任务需要新增“业务层”，优先新增 `com.example.service`（或与现有结构一致的命名），并将复杂业务逻辑下沉到该层。

## 3) Java 21 现代编码规范（必须 + 推荐）

### 3.1 必须
- 语言/构建：以 `todo/pom.xml` 为准（Java 21，Maven）。
- 缩进：4 空格；换行：保持仓库现状（不做无关全局格式化）。
- 命名：
  - 类型：`UpperCamelCase`
  - 方法/变量：`lowerCamelCase`
  - 常量：`UPPER_SNAKE_CASE`
  - 包名：全小写
- 可见性：默认最小化（能 `package-private` 就不要 `public`）。
- import：禁止通配符 `*`；分组 `java.*` / `javax.*` / 第三方 / 本项目；组内字典序。
- 资源释放：I/O/数据库必须 `try-with-resources`；不得泄漏连接与文件句柄。
- 日志：使用 `slf4j` 占位符参数化（禁止字符串拼接）；不得记录敏感信息（密码、token、完整个人隐私数据）。
- 时间：使用 `java.time`（禁止 `Date`/`Calendar` 新代码）。

### 3.2 推荐（现代实践）
- 不可变优先：值对象/DTO 优先用 `record`；集合返回 `List.copyOf/Map.copyOf` 做防御性拷贝。
- `Optional`：只用于返回值，不用于字段/参数；不得用 `Optional.get()` 作为常规流程。
- 异常边界：只在边界层捕获（UI/任务入口/线程边界），业务层抛出语义化异常；同一异常只记录一次（避免重复堆栈污染日志）。
- API 设计：方法输入先校验（`Objects.requireNonNull`/显式校验），失败要可诊断。
- 并发：避免共享可变状态；如需共享，明确同步策略并单元测试覆盖。

## 4) JavaFX / UI 规范（必须）

- 线程：
  - UI 更新只能在 JavaFX Application Thread。
  - 耗时任务（DB/网络/大量计算）不得阻塞 UI 线程；使用 `javafx.concurrent.Task/Service` 或 `CompletableFuture` + 自定义 `Executor`。
- Controller：
  - 事件处理函数保持短小（编排/校验/调用业务层/更新 UI）；不直接写 SQL。
  - 禁止在 Controller 中创建不可控全局状态（`static` 单例、后台线程不释放等）。
- 监听与绑定：新增 listener/binding 必须考虑移除与生命周期，避免内存泄漏。
- 用户提示：对用户展示“可理解的错误信息”；细节写日志而不是弹窗堆栈。

## 5) 数据库与持久化（必须）

- SQL 必须参数化：使用 `PreparedStatement`；禁止字符串拼接用户输入。
- 事务：跨多条写操作必须明确事务边界（要么全成功，要么全失败）。
- 性能：避免在 UI 线程执行查询；大型查询要分页/限流或异步加载。
- 兼容性：数据结构变更要考虑旧数据迁移（必要时补迁移脚本/初始化逻辑）。
- SQLite（若使用）：每次连接需确保外键约束开启（例如 `PRAGMA foreign_keys = ON`）。

## 6) 测试策略（必须 + 推荐）

### 6.1 必须
- 改动可测试时优先补单元测试；不新增明显不可测的代码结构。
- 测试要确定性：避免依赖系统时区/当前时间/随机数；必要时引入 `Clock` 或固定种子。

### 6.2 推荐
- 分层测试：
  - 单元测试：业务与模型（快速、无 IO）
  - 集成测试：数据层（可用临时 SQLite/测试库）
- 防回归：修 bug 必须附带测试复现与覆盖（除非明确不可行并说明原因）。

## 7) 工程化落地（建议逐步引入）

规则不靠“自觉”，靠工具自动执行。建议按优先级逐步引入：

- 格式化：`spotless-maven-plugin`（`google-java-format`）或 Checkstyle（统一风格，避免争论）
- 静态分析：SpotBugs/PMD（避免空指针、资源泄漏、低级错误）
- 依赖治理：依赖更新/安全扫描（例如在 CI 中跑 `mvn -DskipTests verify` + 安全插件）
- 编辑器一致性：`.editorconfig` / `.gitattributes` 统一缩进与行尾（仅在团队需要时落地）

## 8) 交付与 Definition of Done（必须）

最终回复必须包含：
- 改动摘要：做了什么 + 改了哪些文件（用绝对路径）
- 验证方式：给出可复制命令（优先最小范围，再到 `mvn test`）
- 风险与回归点：以及如何回滚/规避（若有）

## 9) 安全与隐私（必须）

- 配置与密钥：
  - 禁止把密码、token、连接串私密字段写入仓库；使用本地配置/环境变量/用户输入，并在文档中说明。
  - 日志中禁止输出敏感信息（密码、token、完整身份证/手机号等）。
- 输入校验：任何来自 UI/文件/数据库/网络的输入都视为不可信；先校验再使用。
- 文件系统：路径拼接必须用 `java.nio.file.Path`/`Paths`，避免路径穿越；写文件前校验目标目录。
- 数据库：所有查询必须参数化（本规范第 5 条）；对错误信息做分级（用户提示 vs 开发日志）。

## 10) 文档与架构决策（推荐）

- 非平凡改动（新增模块、改变分层、引入新依赖、改变数据结构）应同步更新：
  - `ARCHITECTURE.md`（架构/模块边界）
  - `DEV.md`（开发方式/命令/环境）
  - `DATABASE_AND_RUNTIME.md`（数据库/运行时假设）
- 如出现关键权衡（例如“为什么选 SQLite 而不是 X”），建议新增 ADR：`docs/adr/` 下按序号记录（小而清晰）。

## 11) 变更管理（推荐）

- 变更要小步可回滚：一次 PR/一次变更聚焦一个主题，避免“顺手大重构”。
- 代码评审关注点：边界是否清晰、失败路径是否可诊断、是否有测试/验证命令、是否有回归风险说明。

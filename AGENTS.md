# Codex 协作与工程规范（本仓库）

目标：把本文件作为本仓库的主规范入口，用统一、可执行、可审计的方式约束协作、架构、编码、测试、安全与交付。

适用范围：本仓库所有文件与目录。更深层目录中的 `AGENTS.md` 可以做补充或收窄约束，但不得削弱本文件中的安全、数据兼容、测试和交付基线。

规范术语：

- **必须**：不满足即视为规范违规，除非在变更说明中明确豁免原因并获得接受。
- **禁止**：默认不可做，除非有明确的、经说明的例外理由。
- **建议**：强烈推荐；若不采用，应能解释原因与权衡。

## 1. 文档目标、适用范围与优先级

- 本文件同时承担两种角色：
  - Codex / 代理协作约束
  - 仓库工程规范与质量门禁
- 优先级从高到低如下：
  - 运行平台与上层系统 / 开发者指令
  - 更深层目录中的 `AGENTS.md`
  - 本文件
  - 其他说明文档，如 `README.md`、`DEV.md`、`ARCHITECTURE.md`
- 若文档与代码、构建配置或运行配置冲突，以实际代码和构建配置为准，并在同一变更中修正文档。
- 本仓库当前现实基线如下：
  - 根目录是文档与管理层，真正的 Maven 模块在 `todo/`
  - 语言与构建基线为 Java 21 + Maven
  - 当前是 JavaFX 桌面应用，JPMS 模块名为 `com.example`
  - 默认本地数据链路是 SQLite
  - 当前数据层已演进到 `ScheduleItemService` + `SqlScheduleItemRepository` + Stage-B schema
  - `Schedule` 兼容桥接、legacy MySQL 兼容链路与旧版 SQLite schema 痕迹仍存在，但不是默认本地运行主路径
  - 项目仍处于过渡架构期，新改动必须推动边界更清晰，而不是扩大历史耦合

## 2. 代理 / 协作工作方式

- 开始任务前，**必须**先读上下文，优先从代码、配置、测试和现有文档中收集事实，再动手修改。
- 可并行时，**建议**把任务拆分为互斥写范围的实现子任务和只读审计子任务；但是否委派必须服从平台能力、权限和上层约束。
- **禁止**把固定代理数量、固定模型、固定推理强度写成仓库硬要求。协作方式必须能力感知（capability-aware）。
- 若因平台限制、权限限制、或上层策略不能委派，主线程**必须**继续推进，不得把“无法启用子代理”当作阻塞理由。
- 不能委派时，主线程**必须**执行自审计，并至少覆盖以下检查项：
  - `module-info.java` 的 `exports` / `opens` / `requires` 是否被不必要扩大
  - JavaFX UI 线程规则、后台线程生命周期、执行器关闭与资源释放
  - SQL 注入、日志泄漏敏感信息、路径与输入校验
  - 回归影响：API 行为、配置项、数据库结构、测试覆盖与迁移兼容
- 提问规则：
  - 能从仓库中发现的事实，先查再问
  - 只有在高影响、非可发现的产品偏好或权衡未决时才问
  - **禁止**为了偷懒把可发现事实抛给用户确认
- 交付时，**必须**给出已做验证、自审结果与剩余风险，不能只给“已完成”结论。

## 3. 架构边界与分层职责

### 3.1 总体方向

- 当前允许的依赖方向是：启动 / UI -> `application` -> `data` / `config`
- `data` **禁止**依赖 `controller` 或 `view`
- `model` **禁止**反向依赖 UI 包
- 新增复杂业务逻辑时，**必须**优先下沉到 `com.example.application` 中的 service 风格类型，或其下的子包；**禁止**继续向 `MainController` 堆积业务规则
- 新改动必须推动以下方向：
  - 更薄的 `MainController`
  - 更少的 `Schedule` 兼容桥接扩散
  - 更明确的 `ScheduleItem` / Stage-B 数据边界
  - 更少的 UI 直接感知持久化细节

### 3.2 包职责

- `com.example`
  - 应用入口、打包启动器、JavaFX `Stage` 初始化
  - **禁止**承载业务逻辑、SQL 或持久化细节
- `com.example.application`
  - 组合根、应用服务、主题、导航、本地化、提醒、业务编排
  - 当前主业务入口应以 `ScheduleItemService` 体系为准
- `com.example.controller`
  - FXML controller 与 UI 事件编排
  - **禁止**写 SQL、执行重计算、长时间阻塞 UI 线程
- `com.example.view`
  - 视图组件、交互动效、展示逻辑
  - **禁止**直接依赖数据库实现或维护不可控全局状态
- `com.example.model`
  - 领域模型、值对象、规则对象
  - **建议**尽量 UI 无关；若必须使用 JavaFX Property，应限制在当前模型现实所需的最小范围
- `com.example.data`
  - 仓储、连接工厂、schema 管理、SQL 映射，以及 legacy MySQL 的兼容 JDBC 路径
  - **禁止**依赖 UI
- `com.example.config`
  - 配置加载、路径决策、偏好持久化、运行时装配
  - **禁止**承载业务判断

### 3.3 模块边界

- `module-info.java` 中的 `exports`、`opens`、`requires` **必须**最小化。
- **禁止**为了测试便利或临时绕过限制而无理由扩大模块边界。
- 若必须调整模块边界，变更说明中**必须**写明原因、影响面和是否可回收。

## 4. 编码风格与 API 设计

### 4.1 基础编码规则

- 以 `todo/pom.xml` 为准：Java 21、Maven、当前依赖版本。
- 缩进使用 4 空格。
- **禁止**做与任务无关的全局格式化。
- 命名规范：
  - 类型：`UpperCamelCase`
  - 方法 / 变量：`lowerCamelCase`
  - 常量：`UPPER_SNAKE_CASE`
  - 包名：全小写
- 可见性**必须**最小化；能用 `package-private` 的不要用 `public`。
- `import` **禁止**使用通配符 `*`。
- `import` 分组顺序：
  - `java.*`
  - `javax.*`
  - 第三方
  - 本项目
- 组内按字典序排序。

### 4.2 API 设计与数据建模

- 方法入参**必须**先校验；对必填参数使用 `Objects.requireNonNull` 或显式校验。
- 错误信息**必须**可诊断，不能只抛出无上下文的 `RuntimeException`。
- 新增简单不可变载体时，**建议**优先使用 `record`。
- 集合暴露时，**建议**返回防御性拷贝，如 `List.copyOf` / `Map.copyOf`。
- `Optional` 只用于返回值；**禁止**用于字段、序列化模型和方法参数。
- **禁止**把 `Optional.get()` 当作常规流程控制。
- 新代码中的时间处理**必须**使用 `java.time`；**禁止**继续引入 `Date` / `Calendar` 风格 API。
- 公共契约（public contract）包括但不限于：
  - 模块导出接口
  - 配置键与环境变量
  - 持久化 schema 与数据语义
  - 用户可见行为与工作流
- 变更公共契约时，**必须**同步说明兼容性影响、迁移路径和回滚方式。

### 4.3 异常、日志与注释

- 异常**必须**在边界层捕获，如 UI 入口、线程边界、任务入口。
- 业务层与数据层**建议**抛出语义化异常，不要在多层重复记录同一异常堆栈。
- **禁止**无说明地吞异常。确属 best-effort 路径时，必须写出意图且不能掩盖主流程失败。
- 日志**必须**使用 `slf4j` 占位符参数化；**禁止**字符串拼接日志。
- 日志**禁止**输出密码、token、连接串私密字段、完整个人隐私数据、完整任务正文或备注原文，除非有充分脱敏。
- 注释**建议**解释“为什么”与边界条件；**禁止**写纯重复代码表面的注释。

## 5. JavaFX / UI / 并发规则

- 所有 UI 更新**必须**发生在 JavaFX Application Thread。
- 数据库、网络、I/O、重计算**禁止**阻塞 UI 线程。
- 后台任务**必须**使用受控机制，如：
  - `javafx.concurrent.Task` / `Service`
  - `CompletableFuture`
  - 明确生命周期的自定义 `Executor`
- 自定义执行器与后台线程：
  - **必须**命名
  - **必须**有所有者
  - **必须**在关闭路径中释放
- Controller 只负责：
  - 参数收集
  - 基础校验
  - 调用应用服务
  - 更新 UI
- Controller **禁止**直接写 SQL、维护数据库事务、长期持有未释放的后台线程、创建不可控全局状态。
- 新增 listener / binding 时，**必须**考虑解绑、生命周期与内存泄漏。
- 用户可见错误提示**必须**可理解；堆栈与实现细节写入日志，不直接弹给用户。
- 视图层若出现明显业务判断膨胀，**必须**回收至 `application` 层，而不是继续堆在 `view` 或 `controller`。
- 涉及提醒、轮询、自动同步、动画句柄等生命周期对象时，**必须**在应用关闭或页面销毁时释放。

## 6. 数据访问、事务与 schema 演进

### 6.1 当前仓库现实

- 默认本地持久化链路是 SQLite。
- 当前默认 schema 现实以 Stage-B 管理器为准，核心表包括：
  - `schedule_item`
  - `tag`
  - `reminder`
  - `recurrence_rule`
  - `sync_outbox`
  - `sync_checkpoint`
  - `device_registry`
- legacy MySQL 兼容链路与旧版 SQLite 迁移脚本仍存在，但它们不是默认主路径。
- 新代码**禁止**继续假设“唯一真实模型就是旧 `schedules` 单表”。

### 6.2 数据访问规则

- SQL **必须**参数化；使用 `PreparedStatement`，**禁止**拼接用户输入。
- I/O、`Connection`、`Statement`、`ResultSet` **必须**使用 `try-with-resources`。
- 涉及多条写操作时，**必须**显式定义事务边界，并保证全成功或全失败。
- **禁止**在 UI 线程执行数据库查询、迁移或批量写入。
- 数据层**必须**集中处理连接配置、schema 初始化与映射，**禁止**把这些细节扩散到 UI。
- 文件系统路径处理**必须**使用 `Path` / `Paths`，并做 `normalize()` 与目标目录校验，避免路径穿越与误写。

### 6.3 SQLite / schema 演进

- SQLite 连接初始化时，**必须**保留外键约束开启；若修改连接初始化，也必须重新评估 WAL、同步模式和兼容性影响。
- 数据结构变更**必须**同时提供：
  - schema / 初始化逻辑更新
  - 兼容性说明
  - 迁移路径
  - 回滚或修复方案
  - 相关测试
- 修改 `ScheduleItem` 持久化映射、提醒、标签、重复规则或同步字段时，**必须**同步检查仓储映射、测试覆盖与旧模型桥接影响。
- 涉及 legacy 链路的变更，**必须**显式说明为什么要动 legacy，以及是否扩大了其职责。

## 7. 测试策略与质量门禁

### 7.1 基线要求

- 每个非平凡改动**必须**附带验证。
- 可测试的逻辑改动**必须**优先补测试，而不是把不可测结构继续扩大。
- 修 bug 时，**必须**补回归测试；若确实不可行，交付说明中必须写清原因与替代验证方式。
- 测试**必须**尽量确定性：
  - 不依赖当前系统时间
  - 不依赖默认时区
  - 不依赖随机结果
  - 必要时使用固定 `Clock`、固定种子、固定 locale

### 7.2 分层测试建议

- 单元测试：业务规则、模型、格式化、投影与纯逻辑
- 集成测试：仓储、SQLite schema、迁移、配置加载
- UI 测试：在完整 UI 自动化不足时，至少覆盖可分离的状态与辅助逻辑

### 7.3 验证命令与门禁

- 本仓库在 PowerShell / 中文路径环境下，**建议**显式使用 `-f .\pom.xml`。
- 最小验证顺序建议如下：

```powershell
cd .\todo
mvn -f .\pom.xml test
```

- 如需进一步确认构建或启动，再执行：

```powershell
cd .\todo
mvn -f .\pom.xml clean compile
mvn -f .\pom.xml clean javafx:run
```

- 若当前环境缺少 Maven、JDK 或测试前提，交付时**必须**明确说明未执行原因，并给出可直接复制的验证命令。
- 启用中的 CI、静态检查、格式检查或安全检查若已配置，**必须**通过；**禁止**以“本地没跑”为由跳过说明。

## 8. 安全、配置、日志与输入校验

### 8.1 配置与密钥

- 配置优先级遵循当前实现：
  - 环境变量优先于属性文件
  - 属性文件优先级为：`TODO_CONFIG_FILE` 指定路径；否则 `%APPDATA%\ToDo\config\application.properties`；否则工作目录下的 `application.properties`；否则 `config\application.properties`
  - 最后回退到打包进资源的默认值
- **禁止**把密码、token、私密连接串、生产密钥写入仓库。
- 示例配置**必须**使用占位值，不得泄露真实环境信息。
- 新增配置项时，**必须**说明：
  - 键名 / 环境变量名
  - 默认值
  - 生效优先级
  - 对兼容性的影响

### 8.2 输入与输出安全

- 所有来自 UI、文件、数据库、网络或外部配置的输入都视为不可信。
- 输入在进入业务逻辑前**必须**先校验，再使用。
- 错误输出**必须**分层：
  - 用户提示：简洁、可理解、无内部细节
  - 开发日志：有足够上下文，但不泄露敏感信息
- 日志中**建议**记录任务 ID、设备 ID、状态、分类等可排障信息，避免直接输出完整正文或备注。

## 9. 依赖治理、版本管理、兼容性与弃用策略

- 引入新依赖前，**必须**先回答：
  - 现有 JDK、JavaFX 或当前依赖能否解决
  - 新依赖的维护状态、许可证和安全风险如何
  - 是否会扩大包体积、启动成本或迁移成本
- **禁止**为小问题引入重型框架。
- 依赖升级 / 移除 **建议**做成聚焦变更，不与大业务修改混在一起。
- 本仓库当前技术基线默认保持：
  - Java 21
  - 单 Maven 模块
  - JavaFX 21
  - SQLite 默认本地运行链路
- 任何会影响以下内容的变更，**必须**写兼容性说明：
  - 配置键
  - 持久化 schema
  - 模块导出边界
  - 用户可见行为
  - 导入 / 兼容链路
- 弃用（deprecation）规则：
  - 优先先弃用，再删除
  - **必须**说明替代方案与迁移路径
  - **禁止**在同一变更中无说明地移除仍可能被调用、引用或持久化依赖的能力
- 对于仅限内部、无持久化与无公共影响的死代码，可在说明原因后直接删除。

## 10. 代码评审、提交要求与 Definition of Done

### 10.1 变更管理

- 变更**必须**小步、可回滚、聚焦单主题。
- **禁止**把顺手重构、依赖升级、样式整理、架构试验与业务修复混成一个大改动。
- 风险较高的改动，**建议**拆成：
  - 预备重构
  - 功能或修复主体
  - 收尾清理

### 10.2 评审关注点

- 评审时优先关注：
  - 架构边界是否变差
  - Controller 是否继续变重
  - 线程、资源与生命周期是否安全
  - schema、配置与行为是否兼容
  - 日志与错误处理是否可诊断且不泄密
  - 测试是否覆盖真实风险

### 10.3 交付要求

- 最终交付说明**必须**包含：
  - 改动摘要：做了什么、为什么
  - 受影响文件或模块
  - 验证方式：给出可复制命令，优先最小范围，再到完整测试
  - 风险与回归点
  - 回滚或规避方案
- 若涉及以下内容，还**必须**额外写明：
  - 数据变更：迁移、兼容、回滚
  - 配置变更：默认值、优先级、升级说明
  - 用户可见行为变化：预期新行为与受影响路径

### 10.4 Definition of Done

- 满足以下条件方可视为完成：
  - 代码遵守本文件约束
  - 最小必要测试已补齐或说明为何无法补
  - 已执行相应验证，或已明确说明无法执行的客观原因
  - 风险、兼容性与回滚已说明
  - 相关文档已同步更新

## 11. 文档更新与 ADR 规则

- 非平凡改动**必须**同步更新相应文档，按需包括：
  - `README.md`
  - `DEV.md`
  - `ARCHITECTURE.md`
  - `USER.md`
- 文档写作规则：
  - 当前事实与未来规划必须分开写
  - **禁止**把路线图写成已实现事实
  - 若代码已演进，文档必须跟上，不允许长期保留与事实相反的表述
- 出现关键架构或工程决策时，**建议**在 `docs/adr/` 下新增 ADR，至少记录：
  - 背景
  - 决策
  - 备选方案
  - 影响
- 修改本文件时，**必须**同时检查：
  - 是否与当前代码现实一致
  - 是否与上层工具 / 平台约束冲突
  - 是否把建议性条款和阻断性条款混写

本文件的目标不是制造形式主义，而是让贡献者在阅读后能够立即判断：

- 代码应该放在哪一层
- 哪些做法会直接卡评审
- 哪些变更必须补测试或迁移说明
- 如何在当前 JavaFX + SQLite + 过渡架构现实下安全推进项目

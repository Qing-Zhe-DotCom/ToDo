# 阶段 B：数据层可演进化与同步建模

本阶段一句话目标：在已经完成 SQLite 默认化和初步迁移演进的基础上，把当前围绕单机 CRUD 和旧 `Schedule` 模型运转的数据层，继续升级成能够长期支撑全局 ID、完整时间语义、聚合子表、本地优先与后续多设备同步的数据基础设施。

本阶段完成后系统会变成什么状态：客户端的数据层不再围绕 legacy DAO、单表 `schedules` 和本地自增 `int` 主键工作，而是围绕稳定的 Repository、可演进 schema、全局 ID、软删除、聚合子表和同步辅助表运作；后续进入阶段 C 做账号体系和云同步时，不再需要推翻本地模型。

## 目标与完成定义

### 目标

- 完成阶段 B 尚未落地的数据库与数据模型改造。
- 固化未来 2 到 3 年都能承载的本地数据模型。
- 从“已部分落地的分钟级时间字段”升级到更完整的一致时间语义。
- 将当前 `Schedule` 相关存储结构升级为支持全局 ID、软删除、提醒、标签、重复规则和同步辅助表的模型。
- 收口 Repository 边界，继续削弱 legacy DAO 的核心地位。
- 为阶段 C 的 `outbox + checkpoint + push/pull sync` 打好真实可用的本地数据基线。

### 完成定义

- 本地持久化主模型具备全局 UUID、软删除、同步状态和设备来源字段。
- `start_at` / `due_at` 的分钟级基础升级为一致的时间模型，并明确是否全天、精度和时区语义。
- 标签、提醒、重复规则已经拆成独立表或独立聚合结构，不再靠单字段拼接字符串长期承载。
- Repository 层职责稳定，UI 和 Service 不直接感知底层多表结构。
- 当前 SQLite 用户数据可通过迁移脚本平稳进入新模型。
- 阶段 C 依赖的阻塞项全部完成，不需要再返工本地 schema。

### 范围内

- 领域模型升级。
- SQLite 表结构重构与迁移。
- Repository 收口与兼容适配。
- 同步必需字段和本地同步辅助表设计。

### 范围外

- 实际云端同步。
- 账号、权限和工作区后台。
- 商业订阅与计费。

## 当前起点与现实判断

### 当前已经完成的部分

- 客户端默认本地运行基线已经稳定切到 SQLite。
- SQLite migration 框架已经落地，并由 `SqliteMigrationRunner` 驱动。
- 当前已存在：
  - `V001__create_schedules.sql`
  - `V002__add_local_sync_columns.sql`
  - `V003__add_minute_precision_schedule_times.sql`
- 当前主表 `schedules` 已经具备：
  - `deleted_at`
  - `version`
  - `sync_status`
  - `last_synced_at`
  - `device_id`
  - `metadata_json`
  - `start_at`
  - `due_at`
- `ScheduleService -> ScheduleRepository -> SqliteScheduleRepository` 已经是默认本地运行主链路。

### 当前部分完成的部分

- 时间模型已经从“只有 `start_date` / `due_date`”升级到同时支持 `start_at` / `due_at`。
- 但当前只完成了“开始 / 截止”这一层，尚未统一：
  - 结束时间
  - 完成时间
  - 全天任务
  - 时间精度
  - 时区语义
- 同步字段已经进入本地主表，但当前只是基础预留：
  - 还没有 `sync_outbox`
  - 还没有 `sync_checkpoint`
  - 还没有 `device_registry`
  - 还没有真正的同步器
- `deleted_at` 字段虽然已经存在，但当前删除行为仍是物理删除，不是完整 soft delete。

### 当前尚未完成的部分

- 当前主键仍是本地自增 `INTEGER`。
- 当前主模型仍是 `Schedule`，未升级到更长期的规范主实体。
- 当前仍是单表 `schedules`，没有 `schedule_item`、`reminder`、`tag`、`schedule_item_tag`、`recurrence_rule`。
- 当前 `tags` 仍是单字符串，`reminder_time` 仍是单字段。
- 当前 `ScheduleRepository` 仍围绕旧模型和单表接口设计。

### 当前最准确的结论

阶段 B 不是“完全未开始”，而是：

- SQLite 默认化：已完成
- 迁移框架：已完成
- 本地同步辅助字段基础：已完成
- 分钟级开始 / 截止时间基础：已完成
- 完整时间语义：部分完成
- UUID、聚合主模型、子表拆分、同步辅助表、Repository 聚合化：未完成

## 目标状态与固定决策

### 固定决策

- 长期主键固定采用 `UUID` 字符串，不继续使用自增整数作为长期业务标识。
- 长期主实体命名固定为 `ScheduleItem`，迁移期允许保留旧 `Schedule` 作为 UI 兼容模型。
- 主表统一采用软删除，使用 `deleted_at_utc` 作为 tombstone 基础。
- 同步状态和版本字段必须进入主表，而不是只存在内存态。
- 核心字段不能偷懒塞到 JSON；`metadata_json` 只承载扩展属性。
- 分钟级时间是强要求，但“分钟级时间已部分落地”不代表本阶段可以跳过完整时间语义建模。

### 目标主模型

- 目标规范主实体：`ScheduleItem`
- 迁移期兼容展示模型：`Schedule`
- 目标主表：`schedule_item`
- 目标子表：
  - `reminder`
  - `tag`
  - `schedule_item_tag`
  - `recurrence_rule`
- 目标同步辅助表：
  - `sync_outbox`
  - `sync_checkpoint`
  - `device_registry`

### 2. 时间语义决策

本阶段必须把时间语义从“已经有 `start_at` / `due_at`”提升到“整套可解释模型”：

- `start_at`：开始时间
- `end_at`：结束时间
- `due_at`：截止时间
- `completed_at`：完成时间
- `is_all_day`：是否全天
- `time_precision`：当前是日期级还是分钟级
- `timezone`：显示语义使用的时区

## 迁移策略

### 迁移原则

- 保证现有 SQLite 用户数据可平滑迁移。
- 允许一段时间内 `Schedule` 与 `ScheduleItem` 并存，但长期事实来源必须切到新模型。
- 迁移期优先保证“旧 UI 可继续跑”，而不是一次性重写所有页面。

### 迁移步骤建议

1. 增加新表、新列和 UUID。
2. 将旧 `schedules` 数据搬迁或映射到目标主表。
3. 拆分标签和提醒到子表。
4. 建立同步辅助表。
5. 引入兼容层，把旧 UI 读写桥接到新聚合。
6. 最后逐步收缩旧单表和旧接口。

### 兼容要求

- 当前已有用户库必须可以自动迁移。
- 迁移失败时必须保留可回滚路径。
- 新旧模型并存期间，开发者必须明确：
  - 哪个是事实来源
  - 哪个只是兼容壳层

## 与现有 `Schedule` / `ScheduleRepository` 的兼容策略

### `Schedule`

- 当前 `Schedule` 仍是 UI 广泛使用的模型。
- 阶段 B 中不要求立即彻底删掉 `Schedule`。
- 但必须把它降级为过渡展示 / 兼容模型，而不是长期持久化主实体。

### `ScheduleRepository`

- 当前 `ScheduleRepository` 仍围绕旧模型和单表接口设计。
- 阶段 B 必须明确未来方向：
  - 方案 A：保留聚合式 Repository，但内部彻底切到多表聚合读写
  - 方案 B：拆成更细粒度 Repository，再由应用层聚合
- 无论选哪种，阶段 B 结束时都不能再让上层直接依赖旧单表思维。

### legacy DAO

- 不再新增功能到 `ScheduleDAO`
- `ScheduleDAO` 只允许继续承担 legacy MySQL 兼容桥接职责
- 阶段 B 的所有新增模型能力都必须先落在 SQLite / 新 Repository 主链路

## 风险、回滚与兼容

### 主要风险

- 旧 UI 与新领域模型之间的映射成本可能高于预期
- 时间模型升级后，时间轴、热力图和提醒逻辑都可能需要联动修复
- 子表拆分会增加 Repository 聚合事务复杂度
- 旧测试可能过度绑定旧 `Schedule` 字段命名

### 风险缓解

- 采用“新模型入库，旧模型过渡展示”的分步策略
- 先保证主表 / 子表和 Repository 稳定，再逐页升级 UI
- 对迁移脚本和时间转换工具进行单独测试，不把风险分散到页面层

### 回滚策略

- 如果新模型切换引发严重页面回归，可在短期内保留 `Schedule` 兼容层并回退某些 UI 绑定，但不回退 UUID 与软删除等核心决策
- schema 升级失败时，回退到阶段 A 的 SQLite 备份库

### 兼容要求

- 阶段 B 必须兼容阶段 A 的 SQLite 本地优先架构
- 可短期兼容现有服务方法名和视图调用，但底层事实来源必须转向新表结构

## 对阶段 C 的前置阻塞项

阶段 C 不应在以下事项未完成时进入实现期：

- 全局 UUID 已落地
- 主表 / 子表结构已固定
- 软删除行为已真实落地
- 同步辅助表已建立
- 本地 `sync_status` / `version` / `device_id` 语义已冻结
- Repository 聚合边界已稳定
- 分钟级时间模型已补齐到可同步的程度

若上述任一项未完成，不应开始云端同步实现，否则将导致协议、冲突策略和本地出站队列反复推翻。

# 阶段 B：数据层可演进化与同步建模

本阶段一句话目标：把当前围绕单机 CRUD 和旧 `Schedule` 模型运转的数据层，升级成能够长期支撑分钟级时间、全局 ID、子表拆分、本地优先和多设备同步的数据基础设施。

本阶段完成后系统会变成什么状态：客户端的数据层不再围绕 legacy DAO 和本地自增 ID 工作，而是围绕稳定的 Repository、可演进 schema、分钟级时间字段、同步元数据和清晰的主表/子表边界运作；后续做云端同步时，无需再次推翻本地模型。

## 目标与完成定义

### 目标
- 固化未来 2 到 3 年都能承载的本地数据模型。
- 明确“分钟级时间”为硬要求，并建立统一时间语义。
- 将当前 `Schedule` 相关存储结构升级为支持全局 ID、软删除、提醒、标签、重复规则和同步状态的模型。
- 收口 Repository 边界，削弱 legacy DAO 的核心地位。
- 为阶段 C 的 `outbox + checkpoint + push/pull sync` 打好表结构和接口基线。

### 完成定义
- 本地持久化主模型具备分钟级时间、全局 ID、软删除、同步状态和设备来源字段。
- 标签、提醒、重复规则已经拆成独立表或独立持久化结构，不再靠单字段拼接字符串长期承载。
- Repository 层职责稳定，UI 和 Service 不直接感知底层表结构。
- 测试和 UI 仍能在新模型下平稳运行，不出现明显功能退化。
- 旧 `Schedule` 模型有明确的过渡方案，而不是长期继续堆字段。

### 范围内
- 领域模型升级。
- 表结构重构。
- Repository 收口与兼容适配。
- 同步必需字段和本地同步辅助表设计。

### 范围外
- 实际云端同步。
- 账号、权限和工作区后台。
- 商业订阅与计费。

## 目标架构

### 阶段 B 的数据层目标
- 领域基线：
  - 目标主实体命名为 `ScheduleItem`
  - 迁移期允许保留旧 `Schedule` 作为 UI 兼容适配层
- 持久化基线：
  - 主表：`schedule_item`
  - 子表：`reminder`、`tag`、`schedule_item_tag`、`recurrence_rule`
  - 辅助表：`sync_outbox`、`sync_checkpoint`、`device_registry`
- 服务基线：
  - UI 只依赖 `Service`
  - `Service` 只依赖 `Repository`
  - `Repository` 管理事务和聚合落库
- 时间基线：
  - 所有时间点统一使用 UTC 持久化
  - 单独存 `timezone` 以保留显示语义
  - 以分钟为默认精度，保留 `all_day` 和 `time_precision`

### 决策固定
- 全局 ID 固定使用 `UUID` 字符串，不继续使用自增整数作为长期主键。
- 主表统一采用软删除，使用 `deleted_at_utc` 作为 tombstone 基础。
- 同步状态字段必须进入主表，而不是只存在内存态。
- 未来扩展字段通过 `metadata_json` 承载，但核心字段不得偷懒塞入 JSON。
- 分钟级时间是强要求；如果某任务只有日期语义，也通过 `time_precision=day` 表达，而不是回退为另一套独立模型。

## 接口 / 数据 / 配置改动

### 对领域模型的要求
- 规范主实体命名固定为 `ScheduleItem`。
- 旧 `Schedule` 可作为迁移适配层存在，但不再是长期唯一事实来源。
- `Reminder`、`Tag`、`RecurrenceRule`、`SyncStatus`、`OutboxEntry`、`SyncCheckpoint` 必须进入设计文档和代码结构。

### 对 Repository 的要求
- 推荐拆分为：
  - `ScheduleItemRepository`
  - `ReminderRepository`
  - `TagRepository`
  - `SyncRepository`
- 若实现时仍保留聚合式 `ScheduleRepository`，也必须在内部明确主表/子表读写边界。

### 对时间持久化的要求
- SQLite 阶段统一使用 UTC ISO-8601 字符串落库，例如 `2026-04-05T13:45:00Z`。
- PostgreSQL 阶段映射为 `TIMESTAMPTZ`。
- UI 层显示时间时，统一根据 `timezone` 还原。

### 对配置的要求
- 配置层增加默认时区、默认时间精度、默认日程类型等设置项：
  - `todo.app.default-timezone`
  - `todo.app.default-time-precision`
  - `todo.app.default-item-type`

## 风险、回滚与兼容

### 主要风险
- 旧 UI 与新领域模型之间的映射成本可能高于预期。
- 时间模型升级后，时间轴、热力图和提醒逻辑都可能需要联动修复。
- 子表拆分会增加 Repository 聚合事务复杂度。
- 旧测试可能过度绑定旧 `Schedule` 字段命名。

### 风险缓解
- 采用“新模型入库，旧模型过渡展示”的分步策略。
- 先保证主表/子表和 Repository 稳定，再逐页升级 UI。
- 对迁移脚本和时间转换工具进行单独测试，不把风险分散到页面层。

### 回滚策略
- 如果新模型切换引发严重页面回归，可在短期内保留 `Schedule` 兼容层并回退某些 UI 绑定，但不回退全局 ID 和时间语义决策。
- schema 升级失败时，回退到阶段 A 的 SQLite 备份库。

### 兼容要求
- 阶段 B 必须兼容阶段 A 的 SQLite 本地优先架构。
- 可短期兼容现有服务方法名和视图调用，但底层事实来源必须转向新表结构。

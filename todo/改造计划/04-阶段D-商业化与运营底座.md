# 阶段 D：商业化与运营底座

本阶段一句话目标：在阶段 C 的账号体系、设备体系和单用户多设备同步已经稳定的前提下，为产品补齐多租户、权限、订阅、审计、备份、安全和运营能力，让它具备可持续发布与长期运营的基础。

本阶段完成后系统会变成什么状态：产品不再只是“能同步的桌面工具”，而是具备工作区、角色权限、套餐与配额、日志审计、备份恢复、设备注销、数据导出与删除等能力的可运营系统。

## 目标与完成定义

### 目标

- 建立用户、工作区、成员、角色和订阅之间的稳定关系模型。
- 为商业发布准备最小可行的权限和计费架构。
- 为线上服务补齐日志、审计、备份、恢复、监控和故障排查能力。
- 为隐私保护、账号注销、数据导出和数据删除建立流程。
- 明确哪些能力是 v1 商用必做，哪些延后到 v1.5 / v2。

### 完成定义

- 用户身份、工作区、权限和订阅关系清晰且可执行。
- 关键运营能力具备：日志、监控、告警、备份、恢复、错误追踪。
- 商用发布前检查单完整，团队知道如何上线、回滚、排障。
- 数据导出、删除、设备注销、会话管理都有定义。
- 产品能力边界清晰，避免“同步做完了但无法商用”的断层。

### 范围内

- 多租户和权限。
- 订阅和配额模型。
- 运维、日志、备份、恢复、灰度、回滚。
- 基础安全和隐私流程。

### 范围外

- 完整财务系统。
- 大规模多人实时协作高级特性。
- AI 自动排班或复杂协同编排。

## 当前起点与进入前提

### 当前现实

截至当前仓库，阶段 D 还完全未进入实现态。原因不是缺少想法，而是阶段 C 的前提尚未完成。

### 进入阶段 D 的前提

阶段 D 只能建立在阶段 C 真正稳定完成之后，至少要求：

- 用户身份模型稳定
- 设备身份模型稳定
- 单用户多设备同步稳定
- PostgreSQL 云端 schema 已稳定
- push / pull 协议已冻结
- 本地与云端字段语义一致

如果上述前提未完成，不应提前引入权限、订阅、配额和审计体系，否则会把商业化能力建立在不稳定的同步基础上。

## 目标架构

### 商业化后的系统视图

- 客户端：
  - 本地 SQLite
  - 登录态和设备态
  - 权限感知 UI
  - 同步状态与套餐提示
- 服务端：
  - Auth Service
  - Sync API
  - Business API
  - Billing / Subscription Adapter
  - Admin / Ops API
  - PostgreSQL
  - 日志 / 监控 / 告警 / 审计

### 商业化阶段的数据库主线

阶段 D 重点不是重新定义 B/C 的主数据模型，而是建立商业化与运营层表结构：

- `workspace`
- `workspace_member`
- `workspace_invitation`
- `role_permission`
- `plan`
- `subscription`
- `quota_usage`
- `invoice_record`
- `audit_log`
- `security_event`

必须避免把这些表提前混入阶段 B/C，除非只是明确字段预留而非正式上线。

## 多租户与权限主线

### 核心表建议

| 表名 | 作用 | 关键字段 |
| --- | --- | --- |
| `workspace` | 工作区 | `id`, `name`, `type`, `owner_user_id`, `created_at`, `updated_at` |
| `workspace_member` | 工作区成员 | `id`, `workspace_id`, `user_id`, `role`, `status`, `invited_by`, `joined_at` |
| `workspace_invitation` | 邀请记录 | `id`, `workspace_id`, `email`, `role`, `token`, `expires_at`, `status` |
| `role_permission` | 权限映射 | `role`, `permission_key` |

### 固定边界

- 一个 `user` 可以加入多个 `workspace`
- 一个 `workspace` 有多个 `member`
- `member` 具备角色：`owner / admin / editor / viewer`
- 阶段 D 前禁止再出现“只有 `user_id` 没有 `workspace` 边界”的长期核心表

## 测试与验收

### 验收重点

- 用户在不同 workspace 间切换时，看不到不属于自己的数据。
- 免费版用户触发高级能力时，收到明确提示，不是静默失败。
- 设备注销后，被注销设备不能继续同步。
- 数据导出、账号删除和恢复流程有明确执行路径。
- 严重错误能在日志和告警系统中定位。

### 人工验证场景

- 创建个人工作区并邀请第二个成员。
- 为某 workspace 升级套餐，确认能力和限额实时生效。
- 模拟服务端数据库恢复，确认恢复后客户端可以重新同步。
- 管理员查看审计日志，确认成员变更和订阅变更有记录。
- 用户申请数据导出和账号删除，系统能输出清晰状态。

### 失败判定标准

- 工作区、角色和订阅关系仍不清晰。
- 发布和回滚流程依赖人工口头相传。
- 故障发生后无法快速定位日志、恢复备份或关闭问题设备。
- 删除与导出流程没有定义，或者直接造成不可控数据丢失。

## 风险、回滚与兼容

### 主要风险

- 商业化需求过多，容易把阶段 D 做成无限膨胀的大杂烩。
- 权限、订阅和同步边界耦合过深，会导致客户端和服务端逻辑互相污染。
- 运维和审计若后补，数据和故障现场可能无法追溯。

### 风险缓解

- 严格区分 v1、v1.5、v2。
- 将计费、权限、运维和同步分别抽象成独立边界或模块边界。
- 商业发布前完成备份恢复演练和故障演练，而不是只写文档。

### 回滚策略

- 订阅、权限和工作区上线前必须有 feature flag。
- 灰度发布新权限或收费逻辑时，支持快速关闭能力开关。
- 若商业功能异常，不影响客户端本地查看和既有同步能力。

### 兼容要求

- 阶段 D 不得破坏阶段 C 的单用户多设备同步基线。
- 即使商业能力关闭，免费用户仍应继续稳定使用基础本地与同步能力。

## 后续阶段依赖

阶段 D 是四阶段计划的收口阶段，本身不再依赖新的前置阶段，但强依赖阶段 C 已稳定。  
如果以下阻塞项未完成，不应启动商业发布：

- 单用户多设备同步不稳定
- 工作区和角色模型未冻结
- 备份恢复流程未演练
- 审计日志与告警未接入
- 数据导出 / 删除 / 设备注销未定义

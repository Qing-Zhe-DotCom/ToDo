# 阶段 C：云端化与多端同步

本阶段一句话目标：在本地 SQLite 和新数据模型已经稳定的前提下，引入账号体系、云端 PostgreSQL 和 Local-first Sync，让应用从单机桌面版升级为“单用户多设备可同步”的产品。

本阶段完成后系统会变成什么状态：客户端仍然优先读写本地 SQLite，但已经具备登录、设备注册、后台同步调度、增量上传、增量拉取、冲突标记和同步恢复能力；云端通过独立 API/Sync 服务接管认证、用户隔离和权威数据存储。

## 目标与完成定义

### 目标
- 建立独立于桌面客户端的云端服务层。
- 固定云端中心数据库为 PostgreSQL。
- 建立用户、设备、工作区和同步游标模型。
- 实现单用户多设备的增量同步闭环。
- 保持客户端仍为本地优先，不因网络中断而失去可用性。

### 完成定义
- 用户可注册、登录，并在客户端获得稳定登录态。
- 同一用户在两台设备上新增、修改、完成、删除任务后，可以增量同步。
- 离线期间的本地变更可积压在 outbox，网络恢复后自动同步。
- 同步失败会被记录、可重试、可恢复，不会直接破坏本地数据。
- 客户端不直连 PostgreSQL，所有网络行为都经过 API/Sync 服务。

### 范围内
- 云端服务、认证、同步协议、客户端同步调度器。
- PostgreSQL 数据模型和 API 文档骨架。
- 设备注册、用户隔离、幂等处理、冲突处理。

### 范围外
- 多人实时协作。
- CRDT。
- 实时编辑与评论流。
- 团队订阅和商业化后台。

## 目标架构

### 技术基线
- 客户端本地库：SQLite
- 云端中心库：PostgreSQL
- 中间层：独立的 `Backend API / Sync API`
- 服务实现建议：独立新仓库或独立服务，采用 `Java 21 + Spring Boot 3.3+ + Maven`
- 数据迁移建议：云端使用 Flyway 管理 PostgreSQL schema
- 认证方式：JWT Access Token + Refresh Token
- 同步方式：客户端本地出站队列 `outbox` + 增量拉取 `checkpoint`

### 为什么云端固定为 PostgreSQL
- PostgreSQL 更适合作为同步中心库和业务中心库，能更稳定承载：
  - 复杂约束
  - 同步元数据
  - 审计日志
  - JSON 元数据
  - 增量游标和批处理
- 客户端本地库和云端中心库本来就承担不同职责：
  - 本地 SQLite 负责离线、启动速度和用户本地可用性
  - 云端 PostgreSQL 负责权威数据、账号、设备、同步日志和后续商业化
- 阶段 C 不再并行维护“MySQL 云端主路线”，避免实现时再次分叉。

### 总体数据流
1. 用户在客户端进行任何增删改查。
2. UI 写入本地 SQLite。
3. 本地事务同时写 `schedule_item` 及相关子表，并写入 `sync_outbox`。
4. 后台同步器检测网络和登录态，批量读取 `sync_outbox`。
5. 客户端调用 `POST /api/v1/sync/push` 上传本地变更。
6. 服务端校验用户、设备、版本和幂等键，写入 PostgreSQL。
7. 服务端返回处理结果和最新服务端版本。
8. 客户端调用 `POST /api/v1/sync/pull` 按 checkpoint 拉增量。
9. 客户端将远端改动合并入本地 SQLite，并更新 `sync_checkpoint`。

### 阶段 C 固定边界
- 只做单用户多设备同步。
- 不做共享空间多人同时编辑。
- 不做 WebSocket 实时推送，v1 先采用轮询或前后台触发同步。
- 不做复杂 CRDT，冲突策略采用可解释的规则。

### C-05 冲突策略任务

#### 初版固定规则
- 普通字段：最后写入优先，以服务端接收时间和版本为准。
- 完成状态：按最新 `updated_at_utc` 判定。
- 删除：使用 `deleted_at_utc` tombstone，不直接物理删除。
- 重复提交：通过 `idempotencyKey` 去重。
- 冲突记录：写入本地 `sync_status=conflict` 并记录到服务端日志。

### 核心人工验证场景
- A 设备离线创建 3 个任务，联网后自动上传成功。
- B 设备登录同一账号后，拉到 A 设备刚创建的任务。

### 失败判定标准
- 客户端需要直接连 PostgreSQL 才能同步。
- 登录或同步失败会导致本地数据不可用。
- 重复 push 生成重复记录。
- 冲突场景下静默覆盖且无日志。

## 风险、回滚与兼容

### 主要风险
- 客户端与服务端字段语义不一致，导致同步后本地显示异常。
- 初版冲突规则过于简单，可能在边界场景下需要人工介入。
- 网络不稳定时，同步器可能积压大量 outbox 记录。
- 匿名本地数据与登录后账号数据合并不当，会引发重复或误删。

### 风险缓解
- 在阶段 B 就冻结主表/子表语义，阶段 C 不再重新定义字段。
- 先只支持单用户多设备，不进入多人协作。
- 所有同步请求都带幂等键和设备身份。
- 保持客户端本地库为第一事实来源，远端冲突先标记、后处理。

### 回滚策略
- 若云端同步服务严重不稳定，可临时关闭 `todo.sync.enabled`，客户端继续单机运行。
- 若某版本协议不兼容，可通过 `sync protocol version` 拒绝高风险请求，并要求客户端升级。

### 兼容要求
- 阶段 C 不得破坏阶段 A/B 的单机可用能力。
- 即便用户不登录，也应能继续使用本地 SQLite 模式。
